## Trabajo 1 — Historial de cambios de usuarios (auditoría)

**Commit:** `[pendiente — cambios preparados en stage]`

### Problema
No quedaba registro de quién creaba, editaba, activaba/desactivaba, eliminaba o restauraba
usuarios del sistema, ni de qué datos tenía el usuario antes y después de cada cambio.
Tampoco se sabía quién había eliminado un usuario.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `database/migrations/2026_07_12_000000_create_user_historials_table.php` | Nueva tabla `user_historials`: usuario modificado, quién hizo el cambio, acción (creado/actualizado/activado/desactivado/eliminado/restaurado) y snapshots JSON antes/después. |
| `database/migrations/2026_07_12_000001_add_deleteuser_id_to_users_table.php` | Columna `deleteuser_id` en `users`: registra quién eliminó al usuario. |
| `app/Models/UserHistorial.php` | Nuevo modelo. `registrar()` guarda eventos; `snapshot()` arma foto legible del usuario (nombre, email, rol, dirección/unidad desde `mamore`, estado, etc. — nunca la contraseña); accessor `cambios` compara antes/después y devuelve solo los campos que cambiaron. |
| `app/Models/User.php` | Eventos `deleted`/`restored`: al eliminar guarda `deleteuser_id`, invalida remember token, cierra sesiones activas del usuario y registra el evento; al restaurar limpia `deleteuser_id` y registra. Relaciones `deletedBy()` e `historiales()`. |
| `app/Http/Controllers/UsersController.php` | Registra historial en crear, actualizar (marca si cambió la contraseña, sin exponerla) y activar/desactivar (ahora con transacción). Nuevos métodos: `historial($id)` (línea de tiempo paginada, requiere permiso `browse_users`) y `restore($id)` (restaura eliminados, requiere `edit_users`). |
| `resources/views/vendor/voyager/users/historial.blade.php` | Nueva vista: línea de tiempo de eventos por usuario, detalle de campos que cambiaron (antes → después) y botón "Restaurar usuario" para eliminados. |
| `resources/views/vendor/voyager/users/list.blade.php` | Botón "Historial" por usuario en la lista. |
| `resources/views/vendor/voyager/users/edit-add.blade.php` | Botón "Ver historial de cambios" al editar un usuario. |
| `app/Actions/UserHistorialAction.php` | Nueva acción Voyager "Historial" para el BREAD de usuarios. |
| `app/Providers/AppServiceProvider.php` | Registra la acción `UserHistorialAction`. |
| `routes/web.php` | Rutas `GET users/{id}/historial` y `POST users/{id}/restore`. |

### Solución
Cada operación sobre un usuario guarda un snapshot completo y legible (con dirección y
unidad resueltas desde la base `mamore`) antes y después del cambio; la vista de historial
muestra la línea de tiempo y resalta solo los campos que cambiaron. La contraseña nunca se
guarda: solo se indica "Actualizada" cuando fue modificada. Al eliminar un usuario se
registra quién lo eliminó y se cierran sus sesiones; desde el historial se lo puede restaurar.

## Trabajo 2 — Registro de peticiones HTTP del panel (log de auditoría)

**Commit:** `[pendiente — cambios preparados en stage]`

### Problema
No quedaba rastro de las peticiones que los usuarios hacen dentro del panel `/admin`
(quién, desde qué IP, a qué URL, con qué datos), útil para auditoría y diagnóstico.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Http/Middleware/Loggin.php` | Nuevo middleware: registra usuario (id, nombre, email), IP, user agent, URL, método y datos de la petición — excluyendo contraseña y tokens. Funciona también para peticiones sin sesión. |
| `app/Http/Kernel.php` | Alias de middleware `loggin`. |
| `routes/web.php` | Middleware `loggin` aplicado a todo el grupo `admin`. |
| `config/logging.php` | Canal `requests`: archivos diarios `requests-AAAA-MM-DD.log`, conservados 90 días. |

### Solución
Toda petición al panel queda registrada en `storage/logs/requests-*.log` con rotación
diaria y retención de 90 días. Se excluyen del registro la contraseña, `_token` y
`_method`, y las peticiones al monitor `admin/compass`.

**Informe de trabajo — 12/07/2026**
**Responsable:** nachoDev

**Actividad:** Implementación de historial de cambios de usuarios, registro de actividad del panel administrativo y documentación del proyecto.

**Problema detectado:**
El sistema no guardaba registro de los cambios hechos sobre los usuarios (quién creó,

cambiaron, quién hizo el cambio y cuándo. Se agregó un botón "Historial" en la lista de
usuarios y la posibilidad de restaurar usuarios eliminados. Además, toda la actividad del
panel queda registrada en archivos de auditoría diarios (conservados 90 días), y se creó
la carpeta de documentación con la bitácora de trabajo.

**Detalle de archivos modificados:**
1. Base de datos: nueva tabla de historial de usuarios y campo para registrar quién eliminó cada cuenta.
2. Usuarios: registro automático de todos los cambios, con comparación antes/después (la contraseña nunca se guarda).
3. Pantallas: nueva vista de historial con línea de tiempo, botones "Historial" y "Restaurar usuario".
4. Auditoría: registro diario de la actividad del panel administrativo (usuario, IP, acción).
5. Documentación: carpeta `docs/` con bitácora de sesiones e informes.

**Resultado:** Trazabilidad completa sobre las cuentas de usuario: se sabe quién hizo cada
cambio y qué cambió, se pueden recuperar usuarios eliminados, y queda un registro de la
actividad del sistema para auditoría.
