## Trabajo 1 — Límite de subida de archivos configurable desde el sistema

**Commit:** `[pendiente]` — `[pendiente]`

### Problema
El tamaño máximo de los archivos que se suben (5 MB) estaba fijo en el código, tanto en la
validación del servidor como en las pantallas. No se podía cambiar desde la configuración
del sistema. Existía el ajuste `configuracion.fileLimit` en el panel, pero no estaba
conectado a nada: cambiar su valor no afectaba las subidas. Además, no todas las pantallas
de subida validaban el tamaño con el mismo criterio ni con el mismo aviso.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `app/Helpers/helpers.php` | Nuevo. Funciones `file_limit_mb()` y `file_limit_kb()`: leen `setting('configuracion.fileLimit')`; si es un número entero mayor a 0 lo usan, y en cualquier otro caso (vacío, 0, negativo, decimal o no numérico) devuelven el predeterminado de 5 MB. |
| `composer.json` | Se registró `app/Helpers/helpers.php` en `autoload.files` para que las funciones estén disponibles en todo el proyecto. |
| `app/Http/Controllers/EntradasController.php` | `validarArchivos()` ahora arma la regla `max:` con `file_limit_kb()` y el mensaje de error con `file_limit_mb()`, en lugar del `5120` / "5 MB" fijos. Cubre los tres puntos de subida (formulario de correspondencia, documento de respaldo y modal "Agregar archivo"). |
| `resources/views/partials/file-validation-js.blade.php` | Nuevo. Validador de archivos en el cliente (extensión, tamaño y firma real de bytes) con avisos SweetAlert; el límite en MB lo inyecta el helper. Expone `window.validarArchivo()` / `window.validarArchivos()` y valida automáticamente los `input[type=file].imageLength`. Los avisos usan `swalTop()`, que fuerza el `z-index` del SweetAlert por encima de los modales de Voyager. |
| `resources/views/entradas/edit-add.blade.php` | Se quitó el validador de archivos duplicado que estaba embebido en la vista y se reemplazó por la inclusión del validador compartido (`partials.file-validation-js`). |
| `resources/views/partials/modal-dropzone.blade.php` | Al input del modal "Agregar archivo" se le agregó la clase `imageLength` (para que use el validador), se ampliaron los formatos aceptados a JPG/PNG/PDF y se agregó `required` con mensaje en español para que no se pueda subir sin seleccionar un archivo. |
| `resources/views/entradas/read.blade.php` | La zona de carga del documento de respaldo ahora valida cada archivo (tamaño/formato) con el validador compartido antes de agregarlo a la lista; el texto de formatos muestra el límite actual (ej. "máx. 5 MB"). Se incluyó el validador compartido. |
| `resources/views/bandeja/read.blade.php`, `resources/views/personerias/read.blade.php`, `resources/views/reservas/read.blade.php` | Se incluyó el validador compartido para que el modal "Agregar archivo" de esas pantallas valide con el mismo criterio. |

### Solución
Se centralizó el límite de subida en un helper único (`file_limit_mb()` / `file_limit_kb()`)
que lee la configuración del sistema y aplica el predeterminado de 5 MB cuando el valor no
es un entero válido mayor a cero. Ese helper alimenta **tanto la validación del servidor**
(en `validarArchivos()`) **como la validación del cliente** (validador JavaScript
compartido con avisos SweetAlert), de modo que todas las pantallas de subida —formulario de
correspondencia, documento de respaldo y modal "Agregar archivo"— usan el mismo límite y la
misma metodología de aviso. El validador de archivos, que antes estaba duplicado dentro de
una vista, se extrajo a un componente reutilizable.

## Trabajo 2 — Ajustes de la validación en el modal "Agregar archivo" (pruebas)

**Commit:** `[pendiente]` — `[pendiente]`

### Problema
Al probar el modal "Agregar archivo" aparecieron dos detalles: (1) el aviso de archivo muy
pesado se mostraba **por detrás** del modal, quedando tapado; y (2) al presionar "Subir
archivo" sin haber seleccionado nada, el formulario se enviaba igual (guardaba un registro
sin archivo).

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `resources/views/partials/file-validation-js.blade.php` | Se agregó `swalTop()`: fija el `z-index` del contenedor de SweetAlert en 100050 (el tema de Voyager fuerza los modales a `z-index:100001`), para que el aviso siempre aparezca por encima del modal. |
| `resources/views/partials/modal-dropzone.blade.php` | El input de archivo ahora es `required` y muestra el mensaje "Debe seleccionar un archivo para subir." si se intenta enviar vacío. |

### Solución
El aviso de validación se muestra ahora por encima de cualquier modal del sistema, y el
botón "Subir archivo" no envía el formulario hasta que se selecciona un archivo, mostrando
un mensaje claro cuando el campo está vacío.

## Informe para presentación (copiar/pegar en Word)

**Informe de trabajo — 14/07/2026**
**Responsable:** nachoDev

**Actividad:** Hacer configurable, desde la pantalla de configuración del sistema, el tamaño máximo de los archivos que se pueden subir, validándolo de forma uniforme en todas las pantallas.

**Problema detectado:**
El tamaño máximo permitido para subir archivos (5 MB) estaba fijo en el programa y no se
podía cambiar desde la configuración. Aunque existía la opción "Límite de Subida de Archivo
(MB)" en el panel, su valor no tenía ningún efecto. Además, no todas las pantallas de carga
avisaban de la misma manera cuando un archivo era muy pesado o de un formato no permitido.

**Solución aplicada:**
Ahora el sistema toma el límite directamente de la opción de configuración: si allí se
indica un número entero mayor a cero, se usa ese valor; si está vacío o en cero, se aplica
automáticamente el predeterminado de 5 MB. Ese mismo límite se aplica en dos niveles: al
enviar el archivo (control del servidor) y al seleccionarlo en la pantalla (aviso inmediato
al usuario). Se unificó el aviso —con una ventana que muestra el peso del archivo frente al
permitido— en todas las pantallas de subida: el formulario de correspondencia, el aviso de
documento de respaldo y la ventana "Agregar archivo".

**Detalle de archivos modificados:**
1. Configuración central del límite: nueva función que lee la opción del panel y aplica 5 MB por defecto cuando no hay un valor válido.
2. Control del servidor: la validación de archivos usa el límite configurado en lugar de un valor fijo.
3. Pantallas de subida: validación uniforme en el navegador (formato, tamaño y contenido real) con el mismo aviso, reutilizando un único componente.

**Resultado:** El administrador puede cambiar el tamaño máximo de subida desde la
configuración del sistema y ese límite se respeta en todas las pantallas, tanto al
seleccionar el archivo como al guardarlo. Si no se define un valor, el sistema usa 5 MB.
